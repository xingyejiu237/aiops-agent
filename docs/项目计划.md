# AIOps 智能运维助手 — 项目计划

## 一、项目定位

基于 ReAct 推理模式的单 Agent 智能运维助手，解决研发运维团队 OnCall 排障效率问题。
Agent 像一个 SRE 工程师一样，通过"观察→思考→行动"循环，自主完成故障定位与建议生成。

---

## 二、做什么 / 不做什么

### 做什么

| 编号 | 功能 | 说明 |
|------|------|------|
| F1 | ReAct 推理对话 | 用户提问后，Agent 自主多轮 Think→Act→Observe，最终给出结论 |
| F2 | Prometheus 指标查询 | 通过 HTTP API 查询服务 CPU/内存/QPS/延迟等指标 |
| F3 | Grafana Loki 日志查询 | 通过 HTTP API 按服务/关键词/时间范围查询日志 |
| F4 | RAG 知识库检索 | 从运维手册/SOP 文档中检索相关排障方案 |
| F5 | 文档管理 | 上传、删除、列表查看知识库文档（向量化存储） |
| F6 | 服务拓扑查询 | 查询服务间依赖关系，支撑故障传播链路推理 |
| F7 | 多轮上下文记忆 | 同一会话内保持对话上下文，支持追问 |
| F8 | SSE 流式输出 | 前端实时展示 Agent 推理过程和最终回答 |
| F9 | 告警 Webhook 接收 | 接收 Alertmanager 告警，自动触发 Agent 排障 |
| F10 | 排障报告生成 | 将 Agent 推理过程和结论结构化输出 |

### 不做什么

| 编号 | 不做的事 | 原因 |
|------|---------|------|
| X1 | 多 Agent 协同 | 上个项目已验证多 Agent 延迟翻倍；运维推理链需连续性，拆开反而断链 |
| X2 | Plan-Executor 模式 | 排障是探索式推理（每步结果决定下步方向），不是预设计划式执行 |
| X3 | MCP 协议 | 上个项目已验证 MCP 操数据库多此一举；直接用 Spring AI Tool Callback 更轻量可控 |
| X4 | 意图路由 | 单 Agent 不需要意图分类→路由层，省掉一次额外 LLM 调用 |
| X5 | AI 执行运维操作（重启/扩缩容） | 安全红线：Agent 只给建议不执行，避免误操作导致生产事故 |
| X6 | 自动修复/自愈 | 同 X5，执行层需人工确认，本项目只做"诊断"不做"治疗" |
| X7 | 实时监控大盘 | 这是 Grafana 的活，不是 AI 的活，不做重复建设 |
| X8 | 用户认证/权限系统 | 非核心，项目聚焦 AI 能力，用简单密码或跳过 |
| X9 | 分布式部署 | 单机开发项目，不引入 K8s/微服务架构 |
| X10 | 多租户 | 单团队内部使用，不需要租户隔离 |

### 3.2 请求流转

```
用户提问 "订单服务响应变慢"
    │
    ▼
ChatController (SSE 流式)
    │
    ▼
OpsAgent.chat(userMessage, sessionId)
    │
    ▼ Spring AI ChatClient + Tool Callbacks
    │
ReAct 循环 (LLM 自主决策):
  Think: 先查订单服务的CPU和内存指标
  Act:   prometheusQuery("order-service", "cpu_usage")
  Observe: CPU正常，内存92%
  Think: 内存高，查日志看有没有OOM
  Act:   lokiQuery("order-service", "OOM|OutOfMemory", "1h")
  Observe: 发现频繁Full GC日志
  Think: 查知识库有没有JVM调优方案
  Act:   knowledgeSearch("JVM Full GC 调优方案")
  Observe: 找到建议增大堆内存的SOP
  Think: 查服务拓扑，看下游依赖
  Act:   topologyQuery("order-service")
  Observe: 依赖MySQL和Redis，MySQL响应也慢
  Think: 查MySQL慢查询
  Act:   lokiQuery("mysql", "slow_query", "1h")
  Observe: 发现3条慢查询，其中一条全表扫描
  Final Answer: 根因是MySQL慢查询→订单服务阻塞→内存堆积→Full GC
                建议：1.优化SQL加索引 2.临时增大JVM堆内存
    │
    ▼
SSE 流式返回给前端 (Think/Act/Observe/Answer 全程可见)
```

## 七、核心设计

### 7.1 OpsAgent — 单 Agent + ReAct

```java
// 核心伪代码
@Component
public class OpsAgent {

    private final ChatClient chatClient;
    private final SessionManager sessionManager;

    public Flux<String> chat(String userMessage, String sessionId) {
        // 1. 获取会话历史
        List<Message> history = sessionManager.getRecentMessages(sessionId, 20);

        // 2. 构建 ReAct 请求
        return chatClient.prompt()
                .system(OPS_SYSTEM_PROMPT)       // SRE 角色定义
                .messages(history)                // 多轮上下文
                .user(userMessage)                // 当前问题
                .tools(prometheusQueryTool,       // 注册所有 Tool
                       lokiQueryTool,
                       knowledgeSearchTool,
                       topologyQueryTool,
                       mysqlQueryTool)
                .stream()                         // SSE 流式
                .chatResponse()
                .map(resp -> extractContent(resp));
    }
}
```

**关键点**：
- Spring AI 的 Tool Callback 机制天然支持 ReAct：LLM 遇到需要工具的步骤时自动调用，获取结果后继续推理
- 不需要手动实现 Think/Act/Observe 循环，Spring AI 的 `ChatClient` + `tools()` 已经处理
- 单 Agent = 单次 LLM 上下文 = 推理链连贯

### 7.2 Tool 设计 — 每个 Tool 是一个独立的 Spring Bean

```java
@Component
public class PrometheusQueryTool implements Function<PrometheusQueryTool.Request, PrometheusQueryTool.Response> {

    @Description("查询 Prometheus 指标，如 CPU 使用率、内存使用率、QPS、响应延迟等")
    public record Request(
        @Description("服务名称，如 order-service") String service,
        @Description("指标名称，如 cpu_usage, memory_usage, http_requests_total") String metric,
        @Description("时间范围，如 1h, 30m") String duration
    ) {}

    public record Response(String result) {}

    @Override
    public Response apply(Request request) {
        // 调用 Prometheus HTTP API: /api/v1/query_range
        String promql = buildPromQL(request.service, request.metric);
        String result = prometheusService.queryRange(promql, request.duration);
        return new Response(result);
    }
}
```

### 7.3 RAG — 知识库检索增强

```
上传运维文档 → DocumentComponent 分片 → Embedding → 存入 Qdrant
                                                  ↓
Agent 调用 knowledgeSearchTool → Qdrant 相似检索 → 返回 TopK 文档片段 → LLM 生成回答
```

参数调优（可配置）：
- `rag.chunk-size`: 文档切片大小，默认 500 字符
- `rag.top-k`: 检索返回数量，默认 5
- `rag.similarity-threshold`: 相似度阈值，默认 0.5

### 7.4 服务拓扑 — 内存有向图

```java
@Component
public class TopologyComponentImpl implements TopologyComponent {

    // 内存存储服务依赖图，启动时从配置文件加载
    private final DirectedAcyclicGraph<String, DefaultEdge> topology;

    // 支持查询：
    // 1. 查下游依赖：order-service → [mysql, redis, user-service]
    // 2. 查上游依赖：谁依赖了 mysql → [order-service, user-service]
    // 3. 查故障传播路径：mysql 慢 → order-service 受影响 → 前端超时
}
```

拓扑数据来源：`application.yml` 中配置，格式：
```yaml
topology:
  services:
    - name: order-service
      depends-on: [mysql, redis, user-service]
    - name: user-service
      depends-on: [mysql, redis]
    - name: gateway
      depends-on: [order-service, user-service]
```
